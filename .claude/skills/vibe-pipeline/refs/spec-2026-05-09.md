# vibe-pipeline 功能清單(2026-05-09 版本)

> 使用者 2026-05-09 提供的功能規格。已聲明「不是最終結果,目前專案叫做 vibe-pipeline,即使轉 CLI 應該也叫做 vp」。
> 之後會有更新的 ref 取代或補充本檔。

標籤:`[M]` MVP 必要、`[P2]` 第二階段、`[P3]` 未來擴充。

## 1 專案基底

- `[M]` init 命令:在 git repo 內建立 .vibe-pipeline/ 結構(pipelines/、config.json)。Pipeline 一檔一條,tickets 內嵌
- `[M]` `.vibe-pipeline/config.json` project-level 配置:
  - `defaults.doer_runner` / `defaults.critic_runner`
  - `defaults.base_branch`(通常 main)
  - `defaults.branch_pattern`(預設 `pipeline/{slug}`)
  - `defaults.merge_strategy`(預設 squash)
  - `defaults.auto_merge_on_complete`(預設 true)
  - budget 三層 caps + pricing 表
- `[M]` 偵測本機已安裝的 CLI(掃 `$PATH`),只給已裝的選項
- `[P2]` 偵測現有專案技術棧(讀 package.json / pyproject.toml / Cargo.toml)→ 寫入初始 SKILL 草稿
- `[P2]` 掃描既存程式碼結構,產生 SKILL 起手式
- `[P2]` 流程模板庫(web app / CLI tool / library / monorepo …),套用模板 scaffold tickets
- `[P2]` Auth health check:檢查每家 CLI 登入有效性,過期前提醒

## 2 定義層

### Ticket
- `[M]` Ticket schema(YAML):
  - `id, title, status, mode, goal, acceptance, 執行AI, 審核AI?, exclusive?, budget_override?, context_carry?`
- `[M]` Ticket CRUD 命令:`new / edit / show / list / delete`
- `[M]` Ticket schema 驗證(YAML 結構 + cross-ref 完整性)
- `[M]` Mode 切換:pipeline-step ↔ iterative
- `[M]` Mode × Critic 校驗:iterative 必須有 審核AI,否則拒絕儲存
- `[M]` Doer / Critic config(per-ticket):
  - `runner`(claude-cli / codex-cli / …)
  - `prompt`(template,可插 ticket 欄位)
  - 不填則 fallback 到 project defaults
- `[M]` `exclusive` 標記:deploy/DB 類自動標 true
- `[P2]` AC checklist 編輯器(互動式)
- `[P2]` Ticket 範本庫(常見模式,如「加 endpoint」「跑 migration」)

### Pipeline
- `[M]` Pipeline schema(YAML):
  - `id, name, base_branch, branch, steps[], merge_strategy, auto_merge_on_complete, state`
- `[M]` Pipeline CRUD 命令
- `[M]` `steps` 是 ticket id 的有序 list,Pipeline 內部嚴格 sequential
- `[M]` 啟動時自動產生 branch 名稱(用 `branch_pattern`)
- `[M]` Pipeline state machine:`planning / running / paused / completed / failed`
- `[P2]` Pipeline 範本(常見的 ticket 組合)
- `[P3]` Pipeline 階層(子 pipeline 引用)— 第一版不做

### 共通
- `[M]` 列表 / filter:by status、by branch、by age
- `[M]` Status 顯示(含當前 iteration、上次 run 時間)
- `[P2]` Schema migration:工具版本升級時自動 migrate YAML

## 3 Q&A 收斂引擎(cross-cutting)

- `[P2]` Cross-cutting primitive,六個 use sites:
  - `[P2]` Ticket 創建 — 模糊敘述 → goal / AC / prompt / mode
  - `[P2]` AC 定義 — 模糊目標 → 可驗證 AC checklist
  - `[P2]` Critic plan 精修 — 粗略缺失 → 精準下輪 執行AI prompt
  - `[P2]` Pipeline 編排 — 一坨想法 → 排好的 ticket 序列
  - `[P2]` Stall intervention — 卡住時引導使用者修 / 加 prompt
  - `[P2]` SKILL candidate 審核 — AI 提議 → 引導決定納入 / 措辭
- `[P2]` Q&A session 紀錄(可回顧、可復現)
- `[P2]` Q&A 用的 CLI 可獨立指定(跟 執行AI / 審核AI 不一定相同)
- `[P2]` Q&A 中斷與 resume(中途離開可繼續)
- MVP 替代:使用者直接寫 prompt / AC,不走 Q&A,直接 inline 編輯 YAML

## 4 執行

### Doer runner
- `[M]` 構造 執行AI prompt:
  - MVP:單層 = ticket goal + AC + prompt template
  - `[P2]` 三層:SKILL retrieval + pipeline 摘要 + ticket 特定
- `[M]` 呼叫指定 CLI(透過 stdin / arg / file)
- `[M]` 接收 output,擷取:文字回應、token 用量、exit code
- `[M]` 寫入 run log(SQLite)

### Critic runner
- `[M]` 構造 審核AI prompt:執行AI output + AC checklist
  - `[P2]` 加 previous iter verdicts(避免重複指出)
- `[M]` 解析 審核AI 輸出:
  - `verdict: pass | fail`
  - `[M]` `issues: string[]`
  - `[M]` 失敗時的 plan(自由文本)
  - `[P2]` `progress: first | better | same | worse`
- `[M]` 寬鬆 parse(JSON 失敗 → fallback 整段 free text)

### 迭代迴圈
- `[M]` Iterative ticket:loop until pass / abort
- `[P2]` Stall detector:連續 3 輪 progress ∈ {same, worse} → 進 paused
- `[M]` Pipeline-step ticket:單次 執行AI → 審核AI(可選)→ on_fail action
- `[M]` `on_critic_fail` for pipeline-step:`halt`(預設) / `escalate_to_iterative` / `prompt_user`

### Intervention
- `[P2]` 五種介入類型:
  - `edit_prompt` — 重編下一輪 執行AI prompt
  - `append_prompt` — 追加額外指引
  - `redo_qa` — 走 Q&A 重新收斂
  - `override_pass` — 強制標 pass(審核AI 誤判時)
  - `abort` — 終止 ticket
- `[P2]` Intervention log:記在 run log 裡

### Budget tracker
- `[P2]` 累計每張 ticket / pipeline / daily 的 token 與 cost
- `[P2]` 三層軟硬 caps:
  - Ticket level (soft / hard, USD or iter count)
  - Pipeline level (soft / hard)
  - Daily / project level (soft / hard)
- `[P2]` Hard cap 觸發時:當前 ticket 跑完才停,不砍中途
- `[P2]` Pricing table 可使用者更新

### Safety gate
- `[M]` `exclusive: true` ticket 跑時鎖整個系統,其他排隊
- `[M]` Deploy/DB 類 ticket 預設 exclusive
- `[P2]` Dry-run 模式(可選):跑前 simulate 不實際執行
- `[P2]` 破壞性操作確認 prompt(force push、`rm -rf` 等)

## 5 Workspace

### Branch lifecycle
- `[M]` Pipeline 啟動 → `git checkout -b pipeline/{slug} {base_branch}`
- `[M]` 所有 tickets 在該 branch 上跑、commit 累積
- `[M]` Pipeline 完成 → 依 `merge_strategy` merge 回 `base_branch`
- `[M]` Squash / merge / rebase 三種策略支援(預設 squash)
- `[M]` `auto_merge_on_complete` 控制是否自動 merge
- `[M]` Pipeline 失敗 / paused → branch 留著,人可手動上去操作

### Merge 衝突處理
- `[M]` Merge 前先 rebase 到最新 base_branch
- `[M]` 衝突時 → pause + 通知,使用者手動解,resume 後續 merge
- `[P2]` 衝突自動分析:標出哪些檔案 / 行衝突
- `[P3]` AI 輔助衝突解決(審核AI 看 diff 給建議)

### Worktree(opt-in 並行)
- `[P2]` Pipeline 設 `isolated: true` → 啟動時 `git worktree add` 獨立目錄
- `[P2]` 完成後 `git worktree remove`
- `[P2]` 失敗時 worktree 留著,人可上去看
- `[P3]` 多 pipeline 同時跑的並行 scheduler(等實際需要再開)

### 其他
- `[M]` Branch 列表:`pipeline list --active` 列出所有 active pipeline branches
- `[P2]` Branch cleanup:merge 完自動刪 / 留(預設留)
- `[P2]` Stale branch 警示(N 天無動作)

## 6 持續層

### Storage
- `[M]` `.vibe-pipeline/pipelines/<id>.json` — 一檔一 pipeline,tickets 陣列內嵌(git tracked)。id 格式:`<12-hex-ms-ts>-<slug>`
- `[M]` `.vibe-pipeline/config.json` — project config(git tracked)
- `[M]` `.vibe-pipeline/.runtime/runs.db` — SQLite,執行 logs(gitignored)
- `[M]` SQLite tables:`runs`、`iterations`、`interventions`、`budget_ledger`
- `[P2]` SQLite schema migration 機制

### Run log
- `[M]` 每 iteration 紀錄:執行AI output、審核AI verdict、plan、tokens、cost
- `[M]` Pause state 持久化(供 resume)
- `[P2]` Log query 介面(by ticket、pipeline、time range)
- `[P2]` Replay:從 log 重現某次 run 給人看

### Resume / 中斷處理
- `[M]` 工具中斷後重啟 → 偵測 paused tickets,提示 resume
- `[P2]` Crash recovery:中途 OS 死機 → 下次啟動時偵測「running 但孤兒」的 ticket,標為 failed
- `[P2]` 跨 session 的 ticket 接續

### Context retrieval(讀方向)
- `[P2]` SKILL 動態挑片段給 執行AI / 審核AI(不灌全檔)
- `[P2]` 檢索方式:keyword match → embedding similarity(後者後期)
- `[P2]` Pipeline 摘要 propagation:同 pipeline 內後面 ticket 看得到前面
- MVP 替代:執行AI / 審核AI 拿到的就是 ticket 自己的 prompt,沒有額外 context


### Notification
- `[M]` Pipeline 完成 / 失敗:terminal 印出
- `[P2]` Stall warning notification
- `[P2]` Budget threshold notification
- `[P2]` Merge conflict notification
- `[P2]` 通知通道可選:terminal bell / desktop(macOS osascript / Linux notify-send)/ 寫檔給其他工具讀
- `[P3]` Email / Slack webhook 通知

## 階段建議

### MVP([M] 標籤)— 第一個可用版本

最小可動的 ticket 編排器:
- 專案 init + project config
- Ticket / Pipeline schema + CRUD
- Doer + Critic 雙 agent(單層 prompt、無 SKILL retrieval)
- Iterative loop(無 stall detection,只用 max_iter 強制上限)
- Pipeline-step + 可選 審核AI
- Branch-per-pipeline + squash auto-merge
- JSON + SQLite storage
- Exclusive lock(deploy/DB 安全)
- 基本 CLI 命令集

砍掉但留接口的:Q&A 引擎、stall 偵測、budget tracker、worktree、context 三層 prompt、intervention 細節、dashboard。

MVP 的 審核AI 是「死的」:每輪輸出 pass/fail + free-text plan,沒有 progress 判斷,沒有 stall 偵測。max_iter 給個硬上限(例 10)防無限迴圈。Plan 直接塞下一輪 執行AI prompt。

### Phase 2([P2] 標籤)— 主要體驗升級
解決 MVP 用一陣子會痛的點:
- Q&A 收斂引擎(六個 use sites 全開)
- Stall detection + intervention 五種類型
- Budget tracker 三層
- Context 三層 prompt + SKILL retrieval
- Pipeline summary 蒸餾
- Worktree opt-in 並行
- Notification 通道
- Dashboard TUI

### Phase 3([P3] 標籤)— 進階 / nice-to-have
- AI 輔助 merge 衝突解決
- 多 pipeline 真並行 scheduler
- Web UI、Plugin 機制、第三方通知整合

## 設計信條(實作時別丟)

1. **單一定義源** — Ticket / Pipeline 只在 JSON,SQLite 是執行狀態的快取
2. **Branch 是並行邊界** — 不靠 worktree、不靠 lock,靠 git branch 隔離
3. **人工 approve SKILL** — AI 永遠不直接寫 SKILL.md,只能 stage 候選
4. **跨 pipeline 不直傳 context** — 一律走 SKILL 中介
5. **Critic fail 不等於 ticket fail** — Iterative 會重試,Pipeline-step 走 on_fail action
6. **Exclusive lock 永遠優先於並行** — Deploy/DB 不管在哪 branch 都鎖
7. **無 max_iter 預設** — 用 stall detection 替代次數上限,讓 AI 自己決定何時收斂
