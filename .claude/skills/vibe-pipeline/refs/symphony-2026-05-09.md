## 0. 必讀的「狀態警告」

**OpenAI 明說「不會把 Symphony 當產品線維護」**,定位是「low-key engineering preview / reference implementation」。README 第一個 option 直接叫你「拿 SPEC.md 給你的 coding agent 自己 build 一份」。

也就是:
- **SPEC.md(80KB / 2169 行,寫得像 RFC)是核心資產**,Elixir 實作是 reference,不是 canonical
- 22.7k stars / Apache-2.0 / 2026-04-27 釋出 / 5 月後 commit 趨緩
- **抄設計可,期待官方修 bug 不行**

OpenAI 自己提這是 *harness engineering* 概念示範:team 用 coding agent 照 spec 自己生 harness,Symphony 是 reference impl。

## 1. 一句話定位 + 與我們的核心差異

| 維度 | Symphony | vibe-pipeline |
|---|---|---|
| Control plane | **Linear**(tracker)= source of truth | 我們自己的 `.tickets/` YAML + SQLite |
| Agent 角色 | 1:1(issue → 一個 Codex 從頭跑到尾) | doer + critic 二元角色 |
| State 持久化 | **無 DB**,純 in-memory + tracker reconcile | SQLite(planned) |
| Mutate 邊界 | Symphony 只 read tracker;**寫操作全丟給 agent 用 tool 做** | 我們自己寫 ticket / pipeline 狀態 |
| Model 綁定 | 綁 Codex(`codex app-server` stdio) | 走 runner 抽象,多 model |

**他們明確不做的(我們也不做 / 已不一樣)**:
- 沒有 critic 自動審 doer
- 沒有自己的 ticket 系統(完全寄生 Linear)
- 沒有 multi-agent 互相對話(swarm / graph)— 純 supervisor / 1:1
- 沒有 DB(我們有 SQLite)— 走相反方向

## 2. 對齊我們架構的核心抽象(直接借鏡)

### 2.1 `WORKFLOW.md` 模式 — Policy in-repo,單檔多用

repo 裡放一份 `WORKFLOW.md`:
- **YAML front matter** = runtime config(tracker / hooks / codex command / timeouts / max_concurrent_agents)
- **Markdown body** = per-issue prompt template(Liquid 模板)

**一份檔同時做了 dotenv / yaml config / system prompt / playbook 四件事**,改 prompt 就是改 PR,review-able、diff-able、版控。**熱 reload**(改檔不重啟)。

**借鏡點**:
- 我們目前計畫 `.tickets/config.yaml` + `.tickets/skills/*.md` + ticket YAML 的 `prompt` 欄位是分開的
- 可考慮:每個 pipeline 有一份 `pipelines/{name}.md`,front matter 是 config(base_branch / merge_strategy / hooks),body 是 pipeline-level prompt context。doer/critic prompt template 也走同樣模式
- **「Policy in-repo」哲學是核心** — 任何可調設定都該在 repo 內版控,不是 GUI 設了就消失

### 2.2 Run Attempt 11 個明確 lifecycle state

不是 success/fail 二分,而是:
```
PreparingWorkspace → BuildingPrompt → LaunchingAgentProcess →
InitializingSession → StreamingTurn → Finishing →
{Succeeded, Failed, TimedOut, Stalled, CanceledByReconciliation}
```

**借鏡點**:
- 我們的 SQLite `runs.status` 目前 enum 太粗(`running / paused / done / failed / aborted`)
- critic loop 的 retry / 失敗分類**需要這種粒度** — 例如「Stalled」跟「TimedOut」跟「CanceledByReconciliation」要分,UI 才能 surface 不同行動
- 加一層 `iterations.phase` 對應 lifecycle 中的階段(BuildingPrompt / StreamingTurn / Finishing 等),debug 時看 ticket 卡哪一步一目了然

### 2.3 Stall detector 與 turn timeout 分層

三個 timeout 各有不同語意:
- **read timeout**(stdio 沒回應)
- **turn timeout**(單 turn 跑太久)
- **stall timeout**(event 沒進來超過 `stall_timeout_ms` 就 kill + retry)

**借鏡點**:
- 我們之前的 stall detection 設計只談「連續 3 輪 progress same/worse」(語意層 stall)
- Symphony 的 stall 是 **runtime 層**(stdio 沒事件)— 完全不同層次,**兩者都需要**
- 加配置:`config.yaml runtime.timeouts.{read,turn,stall,iter_progress}` 四種 timeout

### 2.4 Reconciliation-driven recovery(取代 durable scheduler state)

每 tick(預設 30s):
- 拉 tracker(Linear)當 source of truth
- 跟 in-memory state 對齊
- 不存 retry timer、不存 session 到 disk
- restart 後**純靠 tracker state + workspace 目錄重建**

代價是 in-flight context 會丟,換到的是「state 永遠對得上 source of truth」強保證。

**借鏡點**:
- 我們有 SQLite 不會走純 reconcile,但 **reconciliation pass 仍應做**:每 tick 校對 SQLite `runs.status='running'` 與實際 process 是否還活著,不一致就 mark orphan/failed
- spec 已提的 `[P2] Crash recovery: 中途 OS 死機 → 偵測「running 但孤兒」的 ticket,標 failed` 就是這個方向,但 Symphony 走得更激進 — 每 tick 都做,不只啟動時做
- git branch 狀態 vs DB 狀態同步也應走 reconcile pattern(branch 沒了 → pipeline 標 failed/cancelled)

### 2.5 Continuation retry vs exponential backoff(兩條 retry 路徑分開)

- **Worker exit 後即使 normal 也排 1s continuation retry**(直到 issue 真的離開 active state)
- **失敗的 retry 用 exponential**:`min(10000 * 2^(n-1), max_retry_backoff_ms)`

兩條路徑語意不同,**不混在一起**。

**借鏡點**:
- 我們的 iterative loop 跟「執行失敗重試」應拆兩條路徑
- iterative pass 是 normal continuation(短間隔,只是繼續推進)
- exec failure(stdio crash / runner 啟不起來)是 exponential backoff
- intervention 五型再加上「continuation policy」:每張 ticket 標明 normal-completion 後要不要 continuation retry

### 2.6 Workspace safety invariants

- `cwd MUST == workspace_path`
- `workspace MUST be under root`
- identifier sanitize 規則(防 path traversal)
- workspace hook 用 `bash -lc` 執行(明確 shell 環境)

**借鏡點**:
- 我們的 git branch / worktree 操作要有同等安全條款
- branch name 走 `pipeline_name` sanitize 規則(只允許 `[a-z0-9-]`,已寫在 spec)
- worktree 路徑 MUST 在 `.tickets/.worktrees/` 底下,絕不允許 `..`
- doer/critic 啟動前先 assert cwd 在 worktree 內,跑出來 abort

### 2.7 Token accounting:絕對 vs 增量分開記

- `last_token_usage` = 增量(這 turn 用的)
- `total_token_usage` = 累積(thread 至今)
- **只信絕對 thread totals,delta 拿來算 diff**

**借鏡點**:
- 我們的 budget tracker 設計需要這層精細度
- `iterations` table 記 `tokens_in / tokens_out`(增量),`runs.total_tokens`(累積)
- ledger 寫入用絕對值,不靠 delta 累加(避免 race / 漏記)

### 2.8 Service / Agent 硬邊界

Symphony 是 **scheduler + reader**;所有 mutate(ticket 寫、PR 開、merge)**全丟給 agent 用 tool 做**(`linear_graphql` client-side tool 把 tracker auth 借給 agent 打 raw GraphQL,但仍是 agent 在做)。

這條邊界讓 orchestrator 變很小、很穩、可測,複雜度全往 prompt + skill 推。

**借鏡點**:
- 我們 backend 的 `Runner` 介面已是這方向(spawn process / 收 stdout / 不做業務)
- **思考:git 操作要不要也丟給 agent?** 例:不要我們自己跑 `git checkout`,而是 doer agent 自己用 git tool 切。Symphony 哲學會說「對」
- 但 vibe-pipeline 的核心信條 #2「Branch 是並行邊界」要求 orchestrator 控 branch — **這是有意的不一樣**。我們要保留 git 操作在 orchestrator(才能保證 isolation),但 PR / merge 後續可考慮丟給 agent

### 2.9 Skills 當 repo-local 工具書

`.codex/skills/{commit, push, pull, land, linear, debug}` 是 repo-local 的 agent SOP,不是 runtime 的一部分,是 prompt 引用的工具書。

**借鏡點**:
- 我們 `.tickets/skills/*.md` 已是這方向
- 補充:**skill 是 prompt 顯式引用,不是塞進 system prompt**(Symphony 強調這點)
- 比 vibe-kanban 的 profile / tag 機制更輕量 — skill 就是 markdown 檔,prompt 用 `{{ skill.commit }}` 之類引用

# Appendix. 原始萃取(由 subagent 整理)

### 一句話定位
把 issue tracker(Linear)當 control plane:每張 ticket 自動分配到一個隔離 workspace、由 Codex agent 端到端跑完(planning → 改 code → 開 PR → 修 review → merge),人只看「狀態欄變化」做高層判斷,不再貼身督導 agent。

### 核心抽象
- **Issue**:tracker 來的工作單;normalized schema(id / identifier / state / labels / blocked_by / priority)。Symphony 自己**不寫 ticket,只讀**
- **WORKFLOW.md**:repo 內版控的「workflow 契約」,YAML front matter = runtime config,Markdown body = per-issue prompt template(Liquid)
- **Workspace**:`<root>/<sanitized-issue-id>/`,每張 ticket 一個目錄,跨 run 保留;agent cwd MUST 是 workspace path
- **Orchestrator**:單一 in-memory 權威狀態,管 `running / claimed / retry_attempts / completed / token totals / rate limits`
- **Agent Runner**:workspace + prompt + Codex app-server client 三件事的封裝
- **Live Session**:`session_id = <thread_id>-<turn_id>`,連續 turn 共用同一 thread
- **Run Attempt**:11 個明確 lifecycle state(見 §2.2)
- **Retry Entry**:scheduled retry,有 attempt / due_at / timer_handle
- **Hooks**:`after_create / before_run / after_run / before_remove`
- **Skills**:`.codex/skills/{commit, push, pull, land, linear, debug}` repo-local agent SOP

### 主要功能模組
- Workflow Loader / Config Layer(typed getter + 熱 reload)
- Tracker Adapter(只有 Linear,介面留 `tracker.kind` 可換)
- Orchestrator(poll / dispatch / retry / reconciliation / 並行配額)
- Workspace Manager(建立、sanitize key、hook 執行、terminal 清理)
- Agent Runner(Codex app-server stdio 接管、turn loop、token 計帳)
- Stall detector(event 沒進來超過 `stall_timeout_ms` 就 kill + retry)
- Optional `linear_graphql` client-side tool
- Optional Phoenix LiveView dashboard + JSON API
- SSH worker(派去遠端 host)

### 運作模型
**Supervisor / scheduler 模型,不是 graph、不是 swarm**。沒有 multi-agent 互相對話,Symphony 是「issue → 一個 agent」1:1 dispatcher。並行靠多 ticket 同時跑(`max_concurrent_agents` 預設 10)。

訊息傳遞:agent 之間**完全隔離**,各自 workspace、各自 Codex thread,不共享 state。唯一共享物是 tracker 狀態(透過 reconciliation 拉回來)。

執行模型:長駐 daemon,每 `interval_ms`(預設 30s)tick 一次。Worker exit 後即使 normal 也排一個 1s continuation retry,直到 issue 真的離開 active state。Backoff: `min(10000 * 2^(n-1), max_retry_backoff_ms)`。

一個 worker 內可跑 multi-turn(`max_turns` 預設 20):第一 turn 用完整 prompt,後續 turn 只送 continuation guidance,**重用同一 thread**。

### 與 model / tool 整合
- **Agent**:綁 Codex(`codex app-server` stdio),官方明說 protocol 以 Codex 版本為準
- **Model**:透過 `codex.command` 字串傳,例 `--config 'model="gpt-5.5"'`
- **Tool 注入**:用 Codex app-server 的 client-side tool 機制,目前只有 `linear_graphql` 一個;其他 tool 由 agent 自己的 MCP / skill 掛
- **Approval / sandbox**:全 pass-through 給 Codex,Symphony 只保證 cwd 不出 workspace root

### 工作流程
1. repo 放 `WORKFLOW.md`(front matter 設 tracker / hooks / codex command;body 寫 prompt)
2. 可選:複製 `.codex/skills/{commit,push,pull,land,linear}` 進 repo
3. Linear team 設好 workflow states(`Todo / In Progress / Human Review / Merging / Rework / Done`)
4. `./bin/symphony ./WORKFLOW.md` 啟動 daemon
5. 在 Linear 把 ticket 從 Backlog 拖到 Todo → Symphony 自己接走、開 workspace、跑 Codex、開 PR、改 review、`land` skill loop 直到 merged → 設 Done
6. 人介入點:寫 ticket / Human Review approve / 出問題改 `WORKFLOW.md`(熱 reload)

### 技術棧
- Elixir 1.19 / OTP 28 / mix(BEAM,選它是為了 supervisor tree 跟 hot reload)
- Phoenix + LiveView + Bandit(可選 dashboard)
- **無 DB**:scheduler state 純 in-memory,recovery 靠 tracker repoll + workspace 目錄
- 透過 `mise` 管 toolchain
- workspace hook 用 `bash -lc`
