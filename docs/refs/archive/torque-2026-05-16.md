## §1 一句話定位 + 核心差異

**Torque** = 本機 single-user multi-agent orchestrator,**桌面為主**(native window via pywebview / 瀏覽器 / iTerm2 Toolbelt 三模式)、用 group/engineer/worker 三層抽象、走 MCP 物理隔離寫權限。

**vs VP 的核心差異**(對齊 / 分歧):

| 維度 | Torque | VP |
|---|---|---|
| 工具限制 | server-side MCP filter(per-agent `.mcp.json` + `X-Torque-Cell-Id` header,daemon filter tool list) | prompt 文字 + Task tool permission(behavioral,sub-agent 自律) |
| Prompt 重用 | Jinja2 action template + transitions DAG | inline ticket.prompt + step/iter/merge mode |
| 平行 | group 內多 worker wave-mode(批 N → 等齊 → review → 派下波) | 跨 pipeline 平行(同 pipeline 線性派 sub-agent) |
| 抽象層 | group → agent → terminal | project → pipeline → ticket |
| 遠端 | 無 mobile / push 設計,純桌面 / 本機 | Tailscale + TOTP + FCM push,有 PWA 移動 |
| 平台 | macOS first | 跨平台(Win/macOS/Linux) |
| Engineer 行為 | watch board + plan wave + dispatch + monitor digest + review diff + merge | runner main agent 線性派 ticket(無 wave / 無 plan) |

共同部分:multi-agent 用 git worktree 隔離、kanban 派 task、orchestrator 角色派 worker、支援 Claude Code + Codex、hooks 拉 event 進 daemon、worktree auto-checkpoint、有 CLI。

## §2 借鏡點(VP 沒有 / 值得抄)

1. **MCP scoping(物理隔離寫權限)** — per-agent 寫 `.mcp.json`(Claude)或 `.codex/config.toml`(Codex)注 `TORQUE_CELL_ID` env,每個 MCP 請求帶 `X-Torque-Cell-Id` header,**daemon 在 server 端 filter tool list 才回 response**。Worker 只看到 `torque_*` 報告工具,engineer 多 `engineer_*`(scoped to own group),architect 多 `architect_*`(per actor)。**沒 override flag,scope is the contract**。
   → VP 當前「主 agent 禁止改 pipeline.json / sub-agent 禁止改 pipeline.json」全靠 prompt 文字約束,sub-agent 物理上仍有 Edit/Write 權,只是它選擇不寫。Phase 6 若要鎖死,MCP scope 路線是真解。

2. **Action = reusable Jinja2 prompt template + transitions DAG** — action 宣告 `transitions`(`feature/implement → feature/review → feature/merge`),這些就是 worker 可呼叫的 `torque_derive` 目標。Task 派 action 時,Torque 渲染 prompt + 插 `torque` context namespace(agent identity / worktree state / task metadata)+ postscript 列出本 action 允許的 MCP tools。
   → VP 的 ticket.prompt 全寫死,常見套路(「建檔 / 改 schema / 跑 test / 寫 doc」)沒法 reuse。如果開始有重複套路,Jinja2 template 化值得抄。

3. **Engineer wave 模式** — engineer 不是「一張 ticket 派一個 sub-agent」,而是「plan next wave → batch N parallel workers → monitor digest → review → 派下波」。
   → VP 的 runner main agent 線性派 sub-agent,平行只發生在跨 pipeline。若 VP 內單 pipeline 要平行多 ticket(獨立的 task),需要 wave 抽象。

4. **Hooks 真即時 event stream** — `SessionStart / PreToolUse / PostToolUse / Stop` 都 POST 到 daemon `/events`,每 tool call / 進度 / session 邊界即時進 daemon。驅動 cell activity badge / engineer digest / worklog / auto-checkpoint trigger。
   → VP 用 fs.watch pipeline.json + notifs.jsonl,polling 性質、粒度只到 ticket status 變動。Torque 那種 tool call 粒度的 stream 是另一個 level,前端 cell 上的活動感才做得到。

## §3 不抄方向

- **macOS-first / iTerm2 Toolbelt 整合 / Ghostty 為下個 target** — VP 是跨平台,terminal-control 強耦合違反 VP 設計
- **Native desktop app(pywebview)** — VP 純 web + PWA mobile,加 native 殼增加維運面但無收益(已有 install 到 home screen)
- **Group 抽象** — VP 的 project → pipeline 兩層已涵蓋,group 是中間層,引入會增加 model 複雜度但用戶心智沒對應實體
- **Inline terminal cell** — VP 不做嵌入式 terminal,使用者用自己慣用 terminal 比較好

## §4 Top 3 借鏡(若只挑三條)

1. **MCP scoping**(權限物理隔離)— 解 VP 當前 prompt 約束的 leak 風險
2. **Hooks event stream → daemon `/events`** — UI 活動感從 ticket 粒度進化到 tool call 粒度
3. **Jinja2 action template + transitions DAG** — 等 VP 有重複 ticket 套路出現再評估
